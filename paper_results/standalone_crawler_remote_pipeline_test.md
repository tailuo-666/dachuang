## 2026-04-09 Bug Fix Verification

### Fix Goal

Restore the intended standalone crawler behavior:

`missing_aspect -> LLM rewrite -> arXiv query generation -> arXiv search`

The previous implementation did call the rewrite stage first, but the prompt template crashed before the LLM request could actually be made.

### Root Cause

In `rag/crawlers/standalone.py`, the `MissingAspectQueryOptimizer` prompt used `ChatPromptTemplate` and embedded an example JSON object directly inside the template:

```text
{
  "original_aspect": "...",
  "optimized_query_en": "...",
  "keywords_en": ["..."]
}
```

For LangChain templates, single `{...}` braces are treated as template variables.
So when `prompt.invoke({"aspect": ...})` ran, LangChain tried to resolve:

- `"original_aspect"`
- `"optimized_query_en"`
- `"keywords_en"`

as input variables instead of plain JSON text.

That caused a `KeyError` before the LLM call.
`rewrite()` then caught the exception and silently fell back to the original aspect, which made the search appear to "skip rewriting".

### Code Fix

The standalone fix changed the JSON example to escaped braces:

```text
{{
  "original_aspect": "...",
  "optimized_query_en": "...",
  "keywords_en": ["..."]
}}
```

This keeps the example JSON as a literal string in the prompt while still allowing:

- `Aspect: {aspect}`

to remain the real template variable.

The fallback behavior was preserved, but it now prints a warning with the aspect and exception information instead of failing silently.

## Why `optimizer.py` Also Has the Same Bug Pattern

The same underlying template problem exists in `rag/query/optimizer.py`.

That file also builds a `ChatPromptTemplate` and embeds unescaped JSON examples such as:

```text
{
  "original_query": "...",
  "normalized_query_zh": "...",
  "retrieval_query_zh": "...",
  "retrieval_query_en": "...",
  "crawler_query_en": "...",
  "keywords_zh": ["..."],
  "keywords_en": ["..."],
  "required_aspects": ["..."]
}
```

and later:

```text
Example:
{
  "original_query": "...",
  ...
}
```

Because these braces are also single `{` and `}`, LangChain will parse keys like `"original_query"` as template variables unless they are escaped.

So the bug is not specific to "missing aspect rewrite" as a product concept.
It is a generic `ChatPromptTemplate` authoring bug:

- any literal JSON inside the template must use `{{` and `}}`
- otherwise LangChain treats JSON keys as placeholders
- then `.invoke(...)` raises a missing-variable error before the LLM is called

### Why We Did Not Fix `optimizer.py` In This Change

This patch intentionally stayed within the standalone crawler scope only.

So the current state is:

- `rag/crawlers/standalone.py`: fixed
- `rag/query/optimizer.py`: still contains the same prompt-template risk and should be fixed separately
